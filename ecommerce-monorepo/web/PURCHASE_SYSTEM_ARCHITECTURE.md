## рҹҺҜ KEY ARCHITECTURAL DECISIONS

### 1. Transaction-Based Inventory Updates
**Decision:** Use database transactions for receiving orders
**Reason:** Ensures data consistency (all-or-nothing updates)
**Benefit:** Prevents inventory discrepancies

### 2. Optimistic UI Updates
**Decision:** Update UI before API confirmation
**Reason:** Faster perceived performance
**Benefit:** Better user experience

### 3. Auto-Generated PO Numbers
**Decision:** System generates sequential PO numbers
**Reason:** Consistency and uniqueness
**Benefit:** Easy tracking and referencing

### 4. Status-Based Workflow
**Decision:** Explicit status progression
**Reason:** Clear process visibility
**Benefit:** Better tracking and reporting

### 5. Snapshot Data in PO Items
**Decision:** Store product name/SKU in PO items
**Reason:** Historical accuracy
**Benefit:** PO remains valid even if product is deleted/changed
